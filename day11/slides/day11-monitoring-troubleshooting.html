System Tables

System Tables provide queryable data about your Databricks account for observability, auditing, and cost analysis.

  • Located in system catalog
  • Historical data retention varies
  • Query with SQL or DataFrames
  • Build custom dashboards
  • Create alerts and reports

Audit Logs (system.access.audit)

SELECT
  event_time,
  action_name,
  user_identity.email AS user_email,
  request_params,
  response.status_code
FROM system.access.audit
WHERE event_date >= current_date() - 7
  AND action_name = 'getTable'
ORDER BY event_time DESC
LIMIT 100;
  • Tracks all user and system actions
  • 90-day retention (default)
  • Essential for compliance

Billing Usage (system.billing.usage)

SELECT
  usage_date,
  workspace_id,
  sku_name,
  usage_type,
  SUM(usage_quantity) AS total_dbus,
  SUM(usage_quantity * list_prices.pricing.default) AS estimated_cost
FROM system.billing.usage
JOIN system.billing.list_prices
  ON usage.sku_name = list_prices.sku_name
WHERE usage_date >= current_date() - 30
GROUP BY 1, 2, 3, 4
ORDER BY estimated_cost DESC;

Cluster Monitoring (system.compute.clusters)

SELECT
  cluster_id,
  cluster_name,
  driver_node_type,
  worker_count,
  state,
  start_time,
  end_time,
  TIMESTAMPDIFF(MINUTE, start_time, COALESCE(end_time, current_timestamp())) AS runtime_minutes
FROM system.compute.clusters
WHERE start_time >= current_date() - 7
ORDER BY start_time DESC;

Job Monitoring (system.workflow)

-- Job success rate by day
SELECT
  DATE(end_time) AS run_date,
  COUNT(*) AS total_runs,
  SUM(CASE WHEN result_state = 'SUCCESS' THEN 1 ELSE 0 END) AS successful,
  SUM(CASE WHEN result_state = 'FAILED' THEN 1 ELSE 0 END) AS failed,
  ROUND(SUM(CASE WHEN result_state = 'SUCCESS' THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) AS success_rate
FROM system.workflow.job_runs
WHERE end_time >= current_date() - 30
GROUP BY 1
ORDER BY 1;

Query Performance (system.query.history)

-- Slowest queries
SELECT
  statement_id,
  executed_by,
  statement_text,
  total_duration_ms / 1000 AS duration_seconds,
  rows_produced,
  warehouse_id
FROM system.query.history
WHERE start_time >= current_date() - 7
ORDER BY total_duration_ms DESC
LIMIT 20;

Identifying Data Skew

Data Skew occurs when data is unevenly distributed across partitions, causing some tasks to take much longer than others.

  • Check Spark UI for task duration variance
  • Look for tasks taking 10x longer than median
  • Common in joins on skewed keys
-- Identify skewed keys
SELECT key_column, COUNT(*) as cnt
FROM table
GROUP BY key_column
ORDER BY cnt DESC
LIMIT 20;

Fixing Data Skew

Adaptive Query Execution
-- Enabled by default in DBR 9.0+
SET spark.sql.adaptive.enabled = true;
SET spark.sql.adaptive.skewJoin.enabled = true;
Salt Keys
-- Add salt to distribute skewed key
SELECT *, CONCAT(key, '_', FLOOR(RAND() * 10)) as salted_key
FROM skewed_table

Memory Troubleshooting

  • OutOfMemoryError – Increase executor memory
  • Excessive spill – More partitions, more memory
  • Driver OOM – Avoid collect() on large data
-- Useful memory configurations
spark.executor.memory = 8g
spark.executor.memoryOverhead = 2g
spark.driver.memory = 8g
spark.sql.shuffle.partitions = 200

Databricks SQL Alerts

  • Create alerts based on SQL query results
  • Trigger on threshold conditions
  • Email, Slack, webhook notifications
  • Schedule evaluation frequency
-- Alert: Job failure rate > 10%
SELECT
  ROUND(SUM(CASE WHEN result_state = 'FAILED' THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) AS failure_rate
FROM system.workflow.job_runs
WHERE end_time >= current_date() - 1
HAVING failure_rate > 10
