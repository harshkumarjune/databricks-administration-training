Speaker Notes

Walk through the evolution. Key point: Lakehouse solves the "two-tier" problem where organizations had to maintain separate lake and warehouse systems.

What is a Lakehouse?

Definition: A Lakehouse is a data management architecture that combines the best features of data lakes (flexibility, scale, low cost) with data warehouses (reliability, performance, governance).

  • Single copy of data for all workloads
  • ACID transactions on data lake storage
  • Schema enforcement and governance
  • Support for BI, ML, streaming, and batch
  • Open formats (Delta Lake, Parquet)

Speaker Notes

Emphasize the "single copy" benefit - no ETL between lake and warehouse. Delta Lake enables ACID on object storage.

Section 2

Control Plane vs Data Plane

Understanding the architecture for administrators

Two-Plane Architecture

Control Plane

Managed by Databricks

  • Web application UI
  • Cluster management
  • Job scheduling
  • Notebook storage
  • Unity Catalog metadata

Data Plane

Runs in Your Cloud

  • Compute clusters (VMs)
  • Data processing
  • Data storage
  • Network resources
  • Your cloud account

Key Insight: Your data never leaves your cloud account. Only metadata flows to the Control Plane.

Speaker Notes

This is critical for security conversations. Customer data stays in their cloud. Databricks only sees metadata and commands.

Control Plane Components

  • Web Application ‚Äì The UI you interact with daily
  • Cluster Manager ‚Äì Orchestrates cluster lifecycle
  • Job Scheduler ‚Äì Manages workflow execution
  • Unity Catalog Service ‚Äì Metadata and governance
  • Repos Service ‚Äì Git integration management
  • Secrets Service ‚Äì Encrypted secret storage

Hosted in Databricks Cloud Account ‚Äì Multi-tenant, highly available, managed by Databricks

Speaker Notes

Control Plane is fully managed SaaS. Admins configure it but don't maintain infrastructure.

Data Plane Components

  • Classic Data Plane ‚Äì Compute in customer's cloud VPC/VNet
  • Serverless Data Plane ‚Äì Databricks-managed compute (optional)
  • Cloud Storage ‚Äì S3, ADLS Gen2, or GCS buckets
  • Network Infrastructure ‚Äì VPCs, subnets, security groups

Classic Compute

  • VMs in your account
  • You control networking
  • Full infrastructure visibility
  • Pay cloud provider for VMs

Serverless Compute

  • VMs in Databricks account
  • No infrastructure management
  • Instant startup
  • Pay Databricks for compute

Speaker Notes

Explain serverless is optional and relatively new. Most enterprises still use classic for network control.

Cloud Architecture: AWS

Control Plane (Databricks)
  • Hosted in Databricks AWS account
  • Communicates via NAT Gateway
  • Manages cluster orchestration
Data Plane (Your AWS)
  • EC2 instances for clusters
  • Customer-managed VPC
  • S3 for data storage
  • IAM roles for access

Cross-Account Access: Databricks uses IAM roles to launch EC2 instances in your account

Speaker Notes

For AWS customers: explain cross-account IAM role trust. This is how Databricks manages resources in customer accounts.

Cloud Architecture: Azure

Control Plane (Databricks)
  • Azure Databricks Service
  • Managed Resource Group
  • Control plane in Databricks tenant
Data Plane (Your Azure)
  • Azure VMs for clusters
  • VNet (injected or managed)
  • ADLS Gen2 for storage
  • Managed identity for access

VNet Injection: Deploy Databricks into your own VNet for network isolation

Speaker Notes

Azure uses managed resource groups. Two resource groups: one you manage, one Databricks manages (contains VMs, disks, NICs).

Cloud Architecture: GCP

Control Plane (Databricks)
  • GCP Databricks Service
  • Managed by Databricks
  • Regional deployment
Data Plane (Your GCP)
  • GCE instances for clusters
  • Customer VPC
  • GCS for data storage
  • Service accounts for access

Private Google Access: Enable for clusters to access Google services privately

Speaker Notes

GCP architecture is similar to AWS. Uses service accounts instead of IAM roles.

Section 3

Databricks Workspace Components

Navigating the platform

What is a Workspace?

Definition: A workspace is a Databricks deployment that provides a unified environment for data engineering, data science, and analytics teams.

  • Contains notebooks, libraries, dashboards, and experiments
  • Manages compute resources (clusters, SQL warehouses)
  • Integrates with cloud storage and external systems
  • Provides access control and collaboration features

Speaker Notes

Think of workspace as a "tenant" or environment. Most orgs have multiple workspaces (dev, staging, prod).

Compute Resources

All-Purpose Clusters
  • Interactive development
  • Shared by multiple users
  • Long-running sessions
  • Manual or auto-termination
Job Clusters
  • Automated workloads
  • Created per job run
  • Terminated after completion
  • Cost-optimized for batch
