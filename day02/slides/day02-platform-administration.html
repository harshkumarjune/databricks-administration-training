Section 1

Account vs Workspace Administration

Two levels of administrative control

Two Levels of Administration

Account Level

Organization-wide settings

  • Workspace management
  • Identity federation (SSO, SCIM)
  • Unity Catalog metastores
  • Billing & subscriptions
  • Network configurations

Workspace Level

Per-workspace settings

  • User & group management
  • Compute policies
  • Workspace features
  • Secret scopes
  • Git integration

Speaker Notes

Most day-to-day admin work happens at workspace level. Account-level changes have broader impact.

Section 2

Workspace Provisioning

Creating and configuring workspaces

Workspace Provisioning Overview

Workspace provisioning creates a new Databricks environment with dedicated resources, configurations, and access controls.

  • Who can provision? Account Admins only
  • Where? Account Console ‚Üí Workspaces ‚Üí Create
  • Requirements: Cloud credentials, network configuration
  • Time: Typically 5-15 minutes

Speaker Notes

Walk through the provisioning process conceptually. We'll do a detailed walkthrough in the lab.

Provisioning Steps

  • Name & Region
    Choose workspace name and cloud region
  • Networking
    Select VPC/VNet or use Databricks-managed
  • Storage
    Configure root storage location
  • Unity Catalog
    Assign metastore (optional at creation)
  • Review & Create
    Validate settings and provision

Speaker Notes

These steps vary slightly by cloud. Focus on the conceptual flow.

Cloud-Specific Provisioning

AWS

  • Cross-account IAM role
  • VPC selection
  • S3 bucket for root storage
  • PrivateLink endpoints (optional)

Azure

  • Resource Group
  • VNet injection or managed
  • ADLS Gen2 storage
  • Private endpoint (optional)

GCP

  • GCP project
  • VPC network
  • GCS bucket
  • Private Google Access

Speaker Notes

Focus on your organization's cloud. The concepts are similar across clouds.

Workspace Naming Conventions

Best Practice: Establish consistent naming conventions before creating workspaces.

Pattern Example Use Case
{org}-{env}-{region} acme-prod-eastus Environment separation
{org}-{team}-{purpose} acme-data-engineering Team isolation
{project}-{env} customer360-dev Project-based

Admin Console Deep Dive

Navigating workspace administration

Admin Console Navigation

Users

Add, remove, and manage workspace users. View entitlements and last login.

Groups

Create and manage groups. Assign users to groups for permission management.

Service Principals

Manage non-human identities for automation and CI/CD.

Cluster Policies

Define allowed cluster configurations to control costs and ensure compliance.

Instance Pools

Pre-provisioned VMs for faster cluster startup times.

SQL Warehouses

Configure and manage SQL compute endpoints.

Speaker Notes

We'll explore each of these in detail throughout the course.

Managing Users

  • Add Users ‚Äì Manually add by email or sync via SCIM
  • Entitlements ‚Äì Workspace access, SQL access, cluster access
  • Admin Assignment ‚Äì Grant/revoke admin privileges
  • Deactivation ‚Äì Remove access without deleting artifacts

‚ö†Ô∏è Best Practice: Use SCIM for automated user lifecycle management. Manual user management doesn't scale.

Speaker Notes

SCIM is covered in Day 8. For now, show manual user management.

Managing Groups

Built-in Groups
  • admins ‚Äì Workspace administrators
  • users ‚Äì All workspace users
Custom Groups
  • Create for teams/projects
  • Use for permissions
  • Sync from IdP via SCIM

Golden Rule: Always assign permissions to groups, never to individual users.

Speaker Notes

Group-based permissions are easier to audit and maintain. Stress this best practice.

Workspace Settings

  • General ‚Äì Workspace name, region (read-only after creation)
  • Advanced ‚Äì Feature toggles, API settings
  • Security ‚Äì Token policies, IP access lists

  • Partner Connect ‚Äì Third-party integrations

Administration Best Practices

  • Plan before provisioning ‚Äì Naming, networking, governance
  • Use groups for permissions ‚Äì Never assign to individuals
  • Implement cluster policies ‚Äì Control costs and standardize
  • Enable audit logging ‚Äì Required for compliance
  • Document everything ‚Äì Configurations, decisions, runbooks
  • Review regularly ‚Äì Users, permissions, policies
