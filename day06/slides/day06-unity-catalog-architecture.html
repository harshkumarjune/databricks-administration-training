Section 1

Unity Catalog Architecture

Understanding the Foundation

What is Unity Catalog?

Unity Catalog is Databricks' unified governance solution for all data and AI assets across clouds and platforms.

  • Centralized governance - Single source of truth for metadata
  • Fine-grained access control - Table, column, and row-level security
  • Automated lineage - Track data flow across transformations
  • Cross-workspace sharing - Share data securely across workspaces
  • AI asset management - Govern models, features, and notebooks

Before vs After Unity Catalog

Before (Legacy Hive Metastore)
  • Per-workspace metadata
  • No cross-workspace sharing
  • Limited access control
  • No lineage tracking
  • Manual auditing
After (Unity Catalog)
  • Centralized metadata
  • Secure data sharing
  • Fine-grained permissions
  • Automatic lineage
  • Built-in audit logging

Section 2

Metastore Configuration

The Foundation of Unity Catalog

The Metastore

Metastore is the top-level container for all Unity Catalog metadata. It stores catalogs, schemas, tables, and access control policies.

  • Region-specific - One metastore per cloud region
  • Account-level resource - Managed at account level, not workspace
  • Shared across workspaces - Multiple workspaces can attach to same metastore
  • Contains root storage - Default location for managed tables
  • Stores all metadata - Objects, permissions, lineage, audit logs

Creating a Metastore

  • Account Admin creates metastore in Account Console
  • Specify name and region
  • Configure root storage location
  • Set up storage credential for root storage
  • Assign Metastore Admin - User or group to manage metastore
  • Enable features - Delta Sharing, Lineage, etc.

Important: A metastore can only be attached to workspaces in the same region. Plan your regional architecture carefully.

Workspace to Metastore Binding

  • Each workspace can only be attached to one metastore
  • Binding is done by Account Admin or Metastore Admin
  • Once bound, workspace users can access Unity Catalog objects
  • Existing hive_metastore data remains accessible (legacy)
-- After binding, users see both:
SHOW CATALOGS;

-- Results:
hive_metastore -- Legacy (per-workspace)
main -- Unity Catalog default
sales_catalog -- Custom catalog

Storage Credentials

Storage Credential = Named object that provides Unity Catalog the ability to access cloud storage using your cloud provider's IAM.

Cloud Credential Type Configuration
AWS IAM Role Role ARN with trust policy
Azure Service Principal or Managed Identity Application ID, Directory ID, Secret
GCP Service Account Service account email

Creating a Storage Credential

-- Create storage credential (AWS example)
CREATE STORAGE CREDENTIAL aws_prod_credential
WITH (
  AWS_IAM_ROLE = 'arn:aws:iam::123456789:role/unity-catalog-role'
)
COMMENT 'Production storage access';

-- Grant usage to data engineers
GRANT CREATE EXTERNAL LOCATION
ON STORAGE CREDENTIAL aws_prod_credential
TO `data_engineers`;

External Locations

External Location = Named object mapping a cloud storage path to a storage credential. Enables governed access to specific storage paths.

  • References storage credential - Uses IAM for access
  • Defines specific path - s3://bucket/path/, abfss://container@account/, gs://bucket/
  • Enables external tables - Tables pointing to existing data
  • Scoped permissions - Grant access to specific locations

Creating an External Location

-- Create external location
CREATE EXTERNAL LOCATION sales_data_location
URL 's3://company-datalake/sales/'
WITH (STORAGE CREDENTIAL aws_prod_credential)
COMMENT 'Sales data external storage';

-- Grant access for creating external tables
GRANT CREATE EXTERNAL TABLE
ON EXTERNAL LOCATION sales_data_location
TO `data_engineers`;
Section 3

Catalogs, Schemas & Tables

Organizing Your Data Assets

Catalogs

Catalog = Top-level container for schemas. Use catalogs to organize data by environment, business domain, or team.

  • Environment separation - dev_catalog, staging_catalog, prod_catalog
  • Domain organization - sales_catalog, hr_catalog, finance_catalog
  • Contains schemas - First level of three-level namespace
  • Permission boundary - USAGE grant required to access
-- Create a catalog
CREATE CATALOG IF NOT EXISTS sales_catalog
COMMENT 'Sales domain data catalog';

Schemas (Databases)

Schema = Container for tables, views, functions within a catalog. Second level of the namespace.

  • Logical grouping - Group related tables together
  • Medallion layers - bronze, silver, gold schemas
  • Default schema - 'default' created automatically in each catalog
  • Managed location - Can specify custom storage path
-- Create schemas for medallion architecture
CREATE SCHEMA sales_catalog.bronze;
CREATE SCHEMA sales_catalog.silver;
CREATE SCHEMA sales_catalog.gold;

Managed vs External Tables

Aspect Managed Table External Table
Data Location Metastore root storage Your specified path
Data Lifecycle Unity Catalog manages You manage independently
DROP TABLE Deletes metadata AND data Deletes metadata only
Use Case New data, full governance Existing data, shared access
Creation CREATE TABLE CREATE TABLE ... LOCATION

Creating Tables

Managed Table
-- Managed table
CREATE TABLE sales.bronze.orders (
  order_id INT,
  customer_id INT,
  amount DECIMAL(10,2),
  order_date DATE
);
External Table
-- External table
CREATE TABLE sales.bronze.legacy_orders (
  order_id INT,
  customer_id INT,
  amount DECIMAL(10,2)
)
LOCATION 's3://bucket/legacy/';

Views in Unity Catalog

  • Standard Views - Saved query, computed at runtime
  • Dynamic Views - Apply row/column security dynamically
  • Cross-catalog views - Can reference tables from multiple catalogs
-- Create a view for analysts
CREATE VIEW sales.gold.order_summary AS
SELECT
  DATE_TRUNC('month', order_date) as month,
  COUNT(*) as order_count,
  SUM(amount) as total_revenue
FROM sales.silver.orders
GROUP BY 1;

Volumes: Governed File Storage

Volumes = Unity Catalog objects for governing access to non-tabular data (files, images, logs, etc.)

  • Managed volumes - Files stored in metastore root storage
  • External volumes - Files in your cloud storage
  • Access control - Same GRANT/REVOKE model as tables
  • Path access - /Volumes/catalog/schema/volume/path
CREATE VOLUME sales.bronze.raw_files
COMMENT 'Raw sales data files';

Data Explorer Interface

  • Catalog Browser - Navigate catalogs → schemas → tables
  • Search - Find tables, columns, and descriptions
  • Schema Details - View column types, descriptions, tags
  • Sample Data - Preview table contents (with permissions)
  • Lineage View - See upstream/downstream dependencies
  • Permissions - View and manage access from UI

Access: Workspace sidebar → Catalog → Data Explorer

Unity Catalog Admin Roles

Metastore Admin

  • Create catalogs
  • Manage storage credentials
  • Configure external locations
  • Grant ownership

Catalog Owner

  • Create schemas
  • Grant catalog access
  • Manage all catalog objects
  • Transfer ownership

Data Owner

  • Create tables/views
  • Grant object access
  • Manage specific objects
  • Define policies

Unity Catalog Best Practices

  • Naming conventions - Consistent, descriptive names (env_domain)
  • Environment separation - Separate catalogs for dev/staging/prod
  • Medallion architecture - Bronze/silver/gold schemas
  • Group-based access - Grant to groups, not individuals
  • Document everything - Use COMMENT on all objects
  • Regular audits - Review permissions quarterly

Common Pitfalls to Avoid

  • Forgetting USAGE - Must grant USAGE on catalog AND schema
  • Wrong region - Metastore must be in same region as workspace
  • Storage credential scope - Too broad = security risk
  • Dropping managed tables - Data is permanently deleted!
  • Legacy hive_metastore - Still exists, may cause confusion
  • Cross-catalog joins - Require permissions on both catalogs
