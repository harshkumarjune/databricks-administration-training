Recommendation: Use SCIM for enterprise environments. Manual management doesn't scale beyond small teams.

Account vs Workspace Users

Account-Level Users
  • Exist in the Databricks account
  • Can be assigned to multiple workspaces
  • Managed via Account Console or SCIM
  • Single source of truth
Workspace-Level Assignment
  • Users must be assigned to workspace
  • Entitlements set per workspace
  • Can have different roles per workspace

Service Principals

Service Principal: A non-human identity used for automation, CI/CD pipelines, and programmatic access.

  • Use Cases: CI/CD, scheduled jobs, API access, integrations
  • Authentication: OAuth 2.0 client credentials
  • Management: Account Console or workspace Admin Console
  • Secrets: Generate OAuth secrets with expiration

Admin Privilege Models

Understanding administrative roles

Admin Roles Hierarchy

Role Scope Key Capabilities
Account Owner Entire Account Full control, billing, cannot be removed
Account Admin Entire Account Workspaces, users, Unity Catalog, SSO
Workspace Admin Single Workspace Users, groups, compute, settings
Metastore Admin Unity Catalog Catalogs, schemas, storage credentials
Data Owner Specific Objects Tables, views they own

Account Admin Capabilities

  • Create and manage workspaces
  • Configure SSO and SCIM at account level
  • Manage account-level users and groups
  • Create and assign Unity Catalog metastores
  • Configure network security (Private Link, IP lists)
  • View billing and usage
  • Manage service principals at account level

Workspace Admin Capabilities

  • Add/remove users from the workspace
  • Create and manage workspace groups
  • Configure cluster policies
  • Manage instance pools
  • Configure SQL warehouses
  • Set workspace-level settings
  • Manage personal access token policies
  • Configure Git integration settings

Granting Admin Access

Account Admin
  • Account Console ‚Üí Users
  • Select user ‚Üí Roles tab
  • Toggle "Account admin"
Workspace Admin
  • Workspace Admin Console ‚Üí Users
  • Select user ‚Üí Edit
  • Check "Admin" checkbox
  • OR: Add to `admins` group

Best Practice: Use the `admins` group for workspace admin assignment. This makes it easy to audit and manage.

Principle of Least Privilege

Definition: Grant users only the minimum permissions necessary to perform their job functions.

  • Start restrictive ‚Äì Add permissions as needed
  • Use roles appropriately ‚Äì Not everyone needs admin
  • Scope permissions ‚Äì Limit to specific resources
  • Review regularly ‚Äì Remove unused permissions
  • Document justification ‚Äì Why does this user need this?

Access Management Best Practices

Implementing secure, scalable access

Access Management Framework

1. Identify

  • Who needs access?
  • What roles exist?
  • What resources?

2. Organize

  • Create group structure
  • Map roles to groups
  • Define entitlements

3. Govern

  • Regular access reviews
  • Audit logging
  • Compliance checks

Example: Group Structure

Group Members Permissions
data-engineers Engineering team members Create clusters, run jobs

Key: Control cluster creation through policies, not by removing entitlement entirely.

Access Reviews

  • Frequency: Quarterly at minimum, monthly for sensitive access
  • What to Review:
    • Admin privileges ‚Äì who has them, do they still need them?
    • Group memberships ‚Äì are they current?
    • Inactive users ‚Äì remove or deactivate
    • Service principal access ‚Äì still needed?
  • Document: Keep records of reviews and decisions

Common Anti-Patterns

‚ö†Ô∏è Avoid these common mistakes

  • Everyone is admin ‚Äì Makes auditing impossible
  • Permissions to individuals ‚Äì Doesn't scale
  • Shared accounts ‚Äì No accountability
  • Stale access ‚Äì Former employees still have access
  • No documentation ‚Äì Why does X have access to Y?
  • Ignoring service principals ‚Äì They're identities too

SCIM Integration (Preview)

SCIM (System for Cross-domain Identity Management) automates user lifecycle management from your Identity Provider.

  • Automatic user provisioning when added to IdP groups
  • Automatic deprovisioning when removed
  • Group membership synchronization
  • Single source of truth for identity
